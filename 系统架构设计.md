# 系统架构设计

## 1. 架构目标

本项目架构服务于三个目标：

1. 快速做出 MVP。
2. 桌宠常驻时足够稳定、低占用。
3. 后续能扩展 Agent Watcher、记忆系统、插件和分享生态。

核心架构：

```text
Electron Main Process
  + Preload 安全桥
  + React Renderer
  + Canvas Pet Engine
  + 内部 Event Bus
  + 分布式状态所有权
```

## 2. 进程模型

```text
Main Process
  - App Shell
  - Window Manager
  - Storage Service
  - DeepSeek Service
  - Reminder Service
  - Agent Watcher
    - AgentEventIngestServer (HTTP)
    - AgentRegistry (Adapter 管理)
    - adapters/ (Claude, OpenClaw, OpenCode, Hermes)
  - Notification Service
  - Share Service

Preload Script
  - contextBridge
  - 白名单 API
  - 事件订阅封装

Renderer Process
  - React UI
  - Canvas Pet
  - AgentLight (状态灯)
  - Chat Bubble
  - Todo Panel
  - Agent Simulator
  - Share Preview
  - Settings Panel
```

边界：

- Renderer 不直接访问 Node.js。
- Renderer 不直接访问文件系统。
- Renderer 不持有 DeepSeek API Key。
- Renderer 不直接操作 electron-store。
- Main Process 不写 UI 组件。
- Preload 不写业务逻辑。

## 4. 事件流

事件流遵循：

```text
Source Adapter
  -> Event Normalizer
  -> Event Bus
  -> State Owner / UI Consumer
```

示例：Agent 等待确认

```text
TraeAdapter
  -> AgentEventNormalizer
  -> agent.waiting_permission
  -> Event Bus
  -> PetEngine
  -> NotificationService
  -> ShareService
```

示例：待办到期

```text
ReminderService
  -> reminder.due
  -> Event Bus
  -> PetEngine
  -> NotificationService
  -> TodoPanel
```

## 5. 状态所有权

本项目不使用巨型全局状态。

```text
Pet Engine owns PetState
Todo Service owns TodoState
Reminder Service owns ReminderState
Agent Watcher owns AgentSessionState
Memory Service owns MemoryState
Chat Service owns ChatSessionState
Share Service owns ShareJobState
Settings Service owns SettingsState
Plugin Runtime owns PluginState
```

规则：

- Event Bus 只做事件通道。
- 业务判断留在状态 owner 模块。
- Renderer store 只保存界面展示状态。
- 跨模块协作用事件或明确 service API。
- 禁止创建 `GlobalStateManager` 或 `AppStateGodObject`。

## 6. 窗口架构

### 6.1 Pet Window

职责：

- 显示桌宠。
- 显示气泡。
- 接收拖拽、点击、右键菜单。

策略：

- 透明无边框。
- 置顶。
- 默认不显示任务栏。
- 窗口尽量小。
- 后续支持鼠标穿透。

### 6.2 Settings Window

职责：

- DeepSeek 配置。
- 桌宠设置。
- 通知设置。
- 记忆管理。
- 插件状态。
- 隐私设置。

策略：

- 按需创建。
- 关闭后销毁或隐藏。
- 不常驻。

### 6.3 Share Window

职责：

- 分享图预览。
- 文案编辑。
- 脱敏确认。
- 保存 PNG。

策略：

- 按需创建。
- 分享图生成不阻塞桌宠窗口动画。

## 7. 鼠标穿透与交互区域

桌宠透明窗口要避免挡住用户操作。

规则：

- 非交互透明区域应尽量鼠标穿透。
- 宠物本体、气泡、右键菜单区域接收点击。
- 拖拽开始时关闭穿透。
- 拖拽结束后恢复穿透策略。
- 气泡展开时扩大交互区域。
- 气泡关闭后缩小交互区域。

MVP 可以先不做精细像素级穿透，但必须保证窗口尺寸足够小，不能大面积挡住桌面。

## 8. 数据流

### 8.1 聊天

```text
Renderer ChatBubble
  -> preload.deskPet.chat.sendMessage
  -> IPC chat:send-message
  -> DeepSeekService
  -> DeepSeek API
  -> response sanitized
  -> Renderer ChatBubble
```

### 8.2 待办

```text
Renderer TodoPanel
  -> preload.todo.create
  -> IPC todo:create
  -> TodoService
  -> StorageService
  -> EventBus todo.created
```

### 8.3 Agent

```text
Agent Source
  -> Agent Adapter
  -> AgentEvent
  -> EventBus
  -> PetEngine / NotificationService / ShareService
```

### 8.4 分享

```text
Agent completed
  -> share.requested
  -> ShareService redaction
  -> SharePreview
  -> User confirms
  -> Save PNG
```

## 9. 依赖规则

允许：

- Renderer 调用 preload 白名单 API。
- Preload 调用 IPC。
- Main service 调用 StorageService。
- Main service 发 EventBus 事件。
- Renderer 订阅安全转发后的事件。

禁止：

- Renderer 直接 import Main 代码。
- Renderer 直接 import Node.js 模块。
- Main 直接操作 React state。
- Service 互相读写内部状态。
- 插件直接操作 UI。
- 模块绕过 EventBus 调用其他模块内部方法。

## 10. MVP 架构验收

完成 MVP 时，架构必须满足：

- main / preload / renderer 三入口清晰。
- 所有 IPC 通道在 `shared/ipcChannels.ts` 注册。
- 所有事件名在 `shared/eventNames.ts` 注册。
- 核心类型在 `shared/types.ts` 定义。
- DeepSeek 只能由 Main Process 调用。
- electron-store 只能由 Main Process 访问。
- Renderer 无 Node.js 权限。
- Agent 状态通过标准 AgentEvent 流转。
