# 赛博宇的桌面AI宠物（产品功能文档）

一个住在桌面上的个人 AI 分身。它有用户自己的形象，能陪用户文字聊天，记住基础偏好，管理待办和提醒，监控 Claude Code、Codex、Trae、小龙虾 OpenClaw 等 AI Agent 的工作状态，并把有趣瞬间生成适合社交媒体传播的内容。

## 1. 项目定位

### 1.1 一句话定位

**赛博宇的桌面AI宠物，是一个陪你工作、监督 AI Agent、提醒你接手任务的桌面 AI 分身。**

### 1.2 核心差异

普通桌宠解决的是“可爱”和“陪伴”，普通 AI 助手解决的是“问答”和“任务执行”。本项目的核心差异是把两者叠加，并进一步绑定 AI Agent 工作流：

- 它不是一个孤立聊天窗口，而是一个可见、可拖拽、会动的桌面角色。
- 它不是只会聊天，而是能理解记忆、待办、提醒、当前任务和 Agent 状态。
- 它不是替代 Claude、Codex、Trae、小龙虾 OpenClaw，而是站在它们旁边，提醒用户什么时候该关注、确认、接手或分享。
- 它不是只服务效率，也服务人格化陪伴和社交传播。

### 1.3 目标用户

首批目标用户建议聚焦在国内的 **AI先锋使用者**：他们不一定是程序员，但会主动尝试新 AI 工具，把 AI 用在工作、创作、学习和个人效率中，并愿意展示自己的 AI 工作流。

- 核心用户：AI先锋使用者，包括开发者、设计师、运营、产品经理、研究者、学生、内容创作者、独立创客等。
- 早期切入场景：使用 Claude Code、Codex、Trae、小龙虾 OpenClaw 等 AI Agent 的用户，因为这类工具天然有“正在运行 / 等待确认 / 已完成 / 失败”的状态，最容易做出差异化提醒。
- 次级场景：用 DeepSeek 聊天、写作、查资料、整理文件、做待办和复盘的日常 AI 用户。
- 后续扩展用户：对虚拟形象、Live2D、像素宠物、桌面伙伴感兴趣的人。

“AI先锋使用者”不是按职业定义，而是按行为定义：

- 愿意尝试新 AI 工具。
- 愿意把 AI 融入日常工作流。
- 愿意用 AI 提升效率或表达自我。
- 愿意把有趣的 AI 使用方式分享到社区。

### 1.4 产品口号

可选口号：

- 让你的 AI 分身住进桌面。
- 它会陪你工作，也会提醒你 Claude 跑完了。
- AI Agent 时代的桌面陪伴层。
- 一个能看着 AI 干活、叫你回来接手的桌宠。

## 2. 产品目标

### 2.1 MVP 目标

第一版不追求大而全，优先验证三个问题：

1. 用户是否愿意让一个“个人形象桌宠”长期停留在桌面上。
2. 用户是否真的需要一个专门提醒 AI Agent 状态的桌面层。
3. 用户是否愿意让桌宠记住基础偏好，并在提醒和聊天中使用。
4. 用户是否愿意把这个桌宠的有趣瞬间分享到社交媒体。

### 2.2 成功标准

MVP 成功可以用以下指标衡量：

- 日活用户中，超过 40% 用户每天打开桌宠超过 2 小时。
- 超过 30% 用户至少配置一个 Agent 监控或待办提醒。
- 超过 25% 用户主动保存至少 3 条记忆。
- 超过 15% 用户使用过一键截图或短视频分享功能。
- 超过 10% 用户愿意导入自定义形象、台词或插件。
- AI先锋社区中产生自然传播，例如 GitHub star、X/即刻/B站/小红书分享、演示视频二创。

## 3. 核心功能

### 3.1 桌宠形象

桌宠是整个产品的第一视觉入口，也是传播的核心资产。

#### 3.1.1 基础能力

- 悬浮在桌面最上层。
- 支持透明窗口。
- 支持鼠标拖拽移动。
- 支持点击、双击、右键菜单。
- 支持贴边、坐在窗口边缘、跟随鼠标、随机走动。
- 支持最小化、隐藏、暂时静音。

#### 3.1.2 形象风格

MVP 可以先支持一种主风格，避免过早分散。

建议优先级：

1. 像素风：开发快，传播感强，适合 GitHub、AI 工具社区和社交媒体传播。
2. Live2D 风：表现力强，适合陪伴感和商业化。
3. 3D VRM 风：沉浸感强，但工程复杂度更高。

首版建议选择：**像素风桌宠 + 后续兼容 Live2D/VRM 的资源接口。**

#### 3.1.3 情绪与动作

首版建议支持 8 个核心状态：

- 待机：正常站立、眨眼、轻微晃动。
- 工作中：盯着屏幕、敲键盘、思考。
- 提醒：举牌、冒泡、挥手。
- 完成：跳跃、鼓掌、庆祝。
- 错误：皱眉、焦虑、提示用户查看。
- 等待确认：举手、盯着用户。
- 休息：坐下、打瞌睡。
- 分享：摆 pose、生成截图。

### 3.2 文字聊天

文字聊天是桌宠人格和 AI 能力的主要入口。

#### 3.2.1 交互方式

- 点击桌宠后弹出小型聊天气泡。
- 气泡跟随桌宠位置，不做成传统全屏聊天软件。
- 支持快捷输入。
- 支持历史对话。
- 支持将聊天结果一键复制。
- 支持把待办、提醒、插件命令从自然语言中提取出来。

#### 3.2.2 大模型接入

首版大模型后台只接 DeepSeek，不做多供应商配置，先保证国内用户能低成本、稳定地完成文字聊天、待办解析和插件意图识别。

- DeepSeek Chat：用于日常聊天、待办解析、记忆提取、分享文案生成。
- DeepSeek Reasoner：后续可用于复杂任务拆解、Agent 结果总结、错误分析。
- 用户只需要配置 DeepSeek API Key。

模型层需要抽象为统一接口：

```text
User message
  -> conversation context
  -> personality prompt
  -> tool/plugin manifest
  -> DeepSeek provider
  -> response
  -> optional tool call
  -> pet reaction
```

后续如果需要支持更多模型，可以保留 Provider 抽象，但不在首版 UI 中暴露多模型选择，避免配置复杂度影响新用户转化。

#### 3.2.3 人格设定

默认人格：赛博宇的桌面 AI 宠物。

人格关键词：

- 有点机灵，但不吵。
- 像工作搭子，不像客服。
- 能提醒、吐槽、鼓励，但不过度打扰。
- 适合 AI先锋使用者、创作者和高频桌面工作场景。

可配置项：

- 名字。
- 称呼用户的方式。
- 说话风格。
- 主动程度。
- 是否允许吐槽。
- 是否允许社交媒体风格输出。

### 3.3 记忆系统

记忆系统是桌宠从“聊天工具”变成“个人 AI 分身”的关键。首版不需要复杂，但必须从一开始把数据结构、权限、检索和升级路线设计好，避免后续长期记忆只能靠聊天记录硬凑。

#### 3.3.1 首版目标

第一阶段的记忆系统只做轻量、可解释、可编辑的短期记忆和基础画像。它是长期能力的底座，但不抢首版 Demo 的主线。

首版目标：

- 记住用户主动告诉它的重要信息。
- 从聊天、待办、Agent 事件中沉淀少量稳定偏好。
- 在对话和提醒中使用这些记忆。
- 所有记忆都能被用户查看、编辑、删除。
- 为后续长期记忆、向量检索、成长系统和个性化推荐留下接口。

首版不做：

- 不自动永久记录所有聊天。
- 不偷偷读取屏幕或文件形成记忆。
- 不做不可解释的黑盒人格推断。
- 不做复杂关系图谱。

#### 3.3.2 记忆类型

建议把记忆拆成 5 类，后续容易扩展：

```text
profile_memory      用户基础画像
preference_memory   用户偏好
task_memory         任务和工作习惯
agent_memory        AI Agent 使用习惯
episodic_memory     重要事件记忆
```

示例：

- 用户基础画像：用户希望被怎么称呼、职业方向、常用语言。
- 用户偏好：喜欢简短提醒、不喜欢过度打扰、偏好中文回答。
- 任务习惯：每天晚上整理 Agent 输出、经常拖延某类任务。
- Agent 习惯：常用 Claude Code，常在权限确认处需要提醒。
- 重要事件：第一次完成项目 Demo、发布了第一条分享视频。

#### 3.3.3 记忆生命周期

记忆不应该只有“存”和“删”，需要有生命周期。

```text
captured
  -> pending_review
  -> active
  -> stale
  -> archived/deleted
```

含义：

- `captured`：系统发现可能值得记住的信息。
- `pending_review`：等待用户确认是否记住。
- `active`：已确认，可用于聊天、提醒、Agent 反馈。
- `stale`：长时间未使用或可能过期。
- `archived/deleted`：归档或删除。

MVP 可以简化为：

```text
active / deleted
```

但数据库字段要预留 `status`，方便后续升级。

#### 3.3.4 记忆写入方式

首版建议支持三种写入方式：

1. 用户显式写入：

```text
“记住，我喜欢你提醒我时说短一点。”
“以后叫我赛博宇。”
“我一般晚上复盘今天的 Agent 输出。”
```

2. 系统建议写入：

```text
桌宠：“我发现你经常在 Claude 等权限确认时离开，要我记住以后重点提醒吗？”
```

3. 插件写入：

```text
Agent Watcher 插件发现用户常用 Claude Code，可提交一条待确认记忆。
Todo 插件发现用户每天固定复盘，可提交一条工作习惯记忆。
```

首版默认只自动保存用户显式写入的记忆；系统建议和插件写入需要用户确认。

#### 3.3.5 记忆读取方式

首版可以先不用复杂向量数据库，而是采用结构化标签 + 简单检索：

- 按记忆类型读取。
- 按标签读取。
- 按最近使用时间读取。
- 按重要性读取。
- 按当前场景读取，例如聊天、提醒、Agent 完成、分享文案。

示例：

```json
{
  "id": "mem_001",
  "type": "preference_memory",
  "content": "用户希望提醒尽量简短，不要连续打扰。",
  "tags": ["reminder", "tone"],
  "importance": 0.8,
  "confidence": 1.0,
  "source": "user_explicit",
  "status": "active",
  "created_at": "2026-05-31T10:00:00+08:00",
  "updated_at": "2026-05-31T10:00:00+08:00",
  "last_used_at": null
}
```

#### 3.3.6 对话中的记忆注入

模型提示词不要塞入全部历史。每次对话只注入少量相关记忆：

```text
Current user message
  -> detect intent and scene
  -> retrieve relevant memories
  -> build compact memory context
  -> model response
  -> optional memory update suggestion
```

记忆上下文示例：

```text
Relevant memories:
- 用户希望被称呼为“赛博宇”。
- 用户喜欢简短提醒，不喜欢连续打扰。
- 用户常用 Claude Code，并希望任务完成后立即提醒。
```

#### 3.3.7 记忆管理界面

设置页需要有一个“记忆”区域：

- 查看当前记忆。
- 按类型筛选。
- 搜索记忆。
- 编辑记忆内容。
- 删除记忆。
- 关闭某类记忆。
- 导出记忆。
- 清空全部记忆。

每条记忆都要显示来源：

- 用户主动告诉。
- 聊天中确认。
- 待办插件建议。
- Agent Watcher 插件建议。

#### 3.3.8 长期扩展方向

后续长期记忆系统可以在这个基础上扩展：

- 向量检索：用于语义相似召回。
- 时间衰减：长期不用的记忆降低权重。
- 记忆合并：把多条重复偏好合并为一条稳定画像。
- 事件时间线：记录重要工作节点和成长事件。
- 关系记忆：记住常见项目、团队成员、客户或朋友。
- 成长系统：桌宠根据长期陪伴解锁动作、台词和人格变化。
- 跨设备同步：Pro 版或账号系统支持。

### 3.4 待办与提醒

待办和提醒是留存功能，不宜首版做得太重。

#### 3.4.1 基础待办

- 新建任务。
- 完成任务。
- 删除任务。
- 设置截止时间。
- 设置提醒时间。
- 设置重复提醒。
- 今日任务列表。
- 桌宠冒泡提醒。

#### 3.4.2 自然语言创建

用户可以直接说：

- “下午三点提醒我看 Claude 跑完没有。”
- “明天早上 10 点提醒我发项目文档。”
- “每天下午 6 点提醒我整理今天的 Agent 输出。”

系统解析为结构化任务：

```json
{
  "title": "看 Claude 跑完没有",
  "remind_at": "2026-05-31T15:00:00+08:00",
  "source": "chat",
  "repeat": null
}
```

#### 3.4.3 提醒呈现

提醒不应只是系统通知，而要体现桌宠感：

- 桌宠走到屏幕边缘。
- 弹出短气泡。
- 播放对应动作。
- 可选系统通知。
- 支持稍后提醒、完成、忽略。

### 3.5 Agent 状态提醒

这是项目最重要的差异化功能。

#### 3.5.1 核心场景

当用户启动 Claude Code、Codex、Trae、小龙虾 OpenClaw、终端任务或其他 AI Agent 后，桌宠负责观察它们的状态：

- 正在运行。
- 正在思考。
- 等待用户输入。
- 等待权限确认。
- 任务完成。
- 任务失败。
- 长时间无输出，疑似卡住。
- 生成了关键文件或结果。

#### 3.5.2 首批支持对象

建议 MVP 支持：

- 首个真实集成对象：优先从 Claude Code 或 Codex 中选择一个打通完整链路。
- 国内用户重点适配：Trae、小龙虾 OpenClaw。
- 普通终端命令：作为兜底监控能力。
- GitHub Actions 或本地 CI 可作为后续功能。

这里的首批支持对象不代表产品只服务程序员，而是选择状态最清晰、演示效果最强的 AI Agent 场景先打穿。后续可以把同一套状态提醒能力扩展到写作 Agent、设计 Agent、浏览器 Agent、资料整理 Agent 和自动化工作流。

#### 3.5.3 状态获取方式

可采用多层策略：

1. 进程监听：识别特定进程是否运行。
2. 窗口标题监听：判断当前 Agent 所在窗口和状态。
3. 日志监听：读取 CLI 输出日志或终端会话记录。
4. 文件监听：监控项目目录变化。
5. 插件主动上报：让 Agent 插件直接把状态发给桌宠。

MVP 优先实现：

- 进程监听。
- 插件主动上报。
- 简单终端状态桥接。

Agent 状态事件建议统一为标准协议，方便后续扩展：

```json
{
  "agent": "trae",
  "session_id": "agent_session_001",
  "status": "waiting_permission",
  "message": "Waiting for user confirmation",
  "project_path": "D:/projects/demo",
  "created_at": "2026-05-31T15:00:00+08:00"
}
```

#### 3.5.4 状态与桌宠反应

```text
Agent started
  -> 桌宠进入工作中状态
  -> 气泡：“我帮你盯着它。”

Agent waiting for permission
  -> 桌宠提醒状态
  -> 气泡：“它在等你确认权限。”

Agent finished
  -> 桌宠庆祝状态
  -> 气泡：“跑完了，要不要我帮你看看结果？”

Agent failed
  -> 桌宠错误状态
  -> 气泡：“这里失败了，我可以帮你总结错误。”

Agent idle too long
  -> 桌宠疑惑状态
  -> 气泡：“它好像卡住一会儿了。”
```

#### 3.5.5 Agent Watcher 插件

首版可以将 Agent 监控作为内置插件：

```text
plugins/
  agent-watcher/
    manifest.json
    index.ts
    README.md
```

插件能力：

- 注册要监听的 Agent 类型。
- 上报运行状态。
- 上报状态变化。
- 请求桌宠展示提醒。
- 请求桌宠触发分享素材生成。

### 3.6 插件式智能

插件系统是后续生态和付费的基础。

#### 3.6.1 插件定位

插件不是简单扩展菜单，而是桌宠的“技能”。

示例：

- Agent Watcher：监控 AI Agent 状态。
- Todo：待办与提醒。
- Calendar：日程读取与提醒。
- Clipboard：剪贴板总结、翻译、改写。
- File Drop：拖文件给桌宠后总结、重命名、分类。
- Screenshot Context：读取当前屏幕截图并分析。
- Web Search：搜索资料并总结。
- GitHub：PR、Issue、Action 状态提醒。
- Dev Server：本地服务启动、崩溃、端口状态提醒。

#### 3.6.2 插件 Manifest

建议插件声明格式：

```json
{
  "id": "agent-watcher",
  "name": "Agent Watcher",
  "version": "0.1.0",
  "description": "Monitor AI agent progress and notify the desktop pet.",
  "permissions": [
    "process:read",
    "filesystem:watch",
    "memory:suggest",
    "notification:send",
    "pet:react"
  ],
  "triggers": [
    "agent.started",
    "agent.waiting_permission",
    "agent.completed",
    "agent.failed"
  ],
  "entry": "index.js"
}
```

#### 3.6.3 插件权限

插件权限需要清晰透明，避免用户担心隐私：

- 文件读取。
- 文件监听。
- 进程读取。
- 网络访问。
- 剪贴板访问。
- 屏幕截图。
- 系统通知。
- 调用 DeepSeek。

用户安装插件时必须看到权限说明。

#### 3.6.4 插件市场

中后期可以做插件市场：

- 免费插件。
- 付费插件。
- 创作者分成。
- 官方精选。
- 企业私有插件。

## 4. 社交媒体病毒式分享

传播功能要从第一版开始设计，不然后期很难自然融入。

### 4.1 分享内容类型

#### 4.1.1 今日工作总结图

自动生成一张图：

- 桌宠形象。
- 今日完成任务数。
- Agent 跑了几次。
- 等待确认几次。
- 最有趣的一句话。
- 用户自定义签名。
- 项目水印。

示例文案：

```text
今天我的桌面 AI 宠物帮我盯了 6 次 Claude，
提醒我接手 3 次，
还吐槽我有 2 个待办从昨天拖到了今天。
```

#### 4.1.2 Agent 完成瞬间

当 Claude Code、Codex、Trae 或小龙虾 OpenClaw 完成任务后：

- 桌宠跳起来庆祝。
- 自动生成 5-10 秒 GIF 或 MP4。
- 带上“Claude 跑完了，我的桌宠比我还激动”之类文案。

#### 4.1.3 个人 AI 分身名片

用户可以生成：

- 我的桌宠形象。
- 我的 AI 人格关键词。
- 我的工作习惯。
- 我的本周 Agent 使用统计。
- 分享二维码或下载链接。

#### 4.1.4 桌宠表情包

自动导出：

- 跑完了。
- 卡住了。
- 等你确认。
- 该休息了。
- 这个待办拖太久了。
- 今天也在监督 AI 打工。

### 4.2 分享入口

分享入口需要低打扰：

- 右键桌宠：生成今日卡片。
- Agent 完成后：生成短视频。
- 每日收尾：是否生成今日总结。
- 任务完成累计达到某个有趣阈值时：生成里程碑图。

### 4.3 分享隐私过滤

分享是增长核心，但 Agent 输出、项目路径、文件名、待办内容可能包含隐私。分享模块必须默认做脱敏和预览。

- 生成前自动识别并隐藏 API Key、Token、邮箱、手机号、用户名、绝对路径。
- 默认不展示完整项目路径，只展示项目名或用户手动填写的展示名。
- 默认不展示完整待办内容，只展示统计数字和用户确认过的短文案。
- 生成前必须预览，用户确认后才能保存或复制。
- 分享模板区分“开发者梗图”和“产品宣传图”，避免所有内容都像广告。

### 4.4 增长飞轮

```text
用户导入个人形象
  -> 形成独特桌宠
  -> 桌宠陪用户工作
  -> Agent 完成/提醒瞬间产生有趣内容
  -> 用户分享到社交媒体
  -> 新用户被“个人 AI 分身 + Agent 监督”吸引
  -> 新用户继续生成自己的桌宠
```

## 6. MVP 范围

### 6.1 首版核心用户路径

第一版必须先跑通一个足够清晰的演示闭环：

```text
1. 用户启动桌宠。
2. 桌宠完成自我介绍，并让用户设置称呼。
3. 用户配置 DeepSeek API Key。
4. 用户和桌宠进行文字聊天，顺手创建一个待办提醒。
5. 用户启动 Claude Code / Codex / Trae / 小龙虾 OpenClaw 中的一个 Agent。
6. 桌宠进入“帮你盯着”状态。
7. Agent 等待确认时，桌宠提醒用户。
8. Agent 完成后，桌宠庆祝并生成分享图。
```

这条路径要同时体现：桌宠可爱、聊天可用、待办有用、Agent 提醒差异化、分享有传播感。

虽然首个演示 Agent 可以是编程工具，但对外表达应避免把产品说成“程序员专用”。更好的表达是：**给 AI先锋使用者的桌面 AI 宠物，先从 AI Agent 状态提醒这个高频痛点切入。**

## 8. 隐私与安全

因为项目会接触屏幕、文件、剪贴板、进程和 AI 对话，隐私必须从第一天设计。

### 8.1 原则

- 默认本地优先。
- 明确展示插件权限。
- 记忆默认本地保存，并允许用户逐条查看和删除。
- 屏幕截图、剪贴板、文件读取必须经过用户授权。
- 不默认上传用户数据。
- 分享内容生成前必须预览。
- 分享内容生成前必须做基础脱敏，避免泄露项目路径、账号、Token 和待办细节。
- 用户可以一键清空聊天、待办、Agent 日志。
- 用户可以一键清空全部记忆。

### 8.2 敏感权限

敏感权限包括：

- 屏幕截图。
- 剪贴板读取。
- 文件系统读取。
- 进程列表读取。
- 窗口标题读取。
- 网络访问。
- 调用 DeepSeek 云端模型。
- 长期记忆同步。

每个插件都应声明权限，并在设置页可见。

## 9. 用户体验设计

### 9.1 首次使用引导

- **桌宠自己引导，不是弹窗教程**。由桌宠通过气泡和动作完成出场、称呼、风格选择和功能演示，不做传统轮播引导页。
- **选择题优于填空题**。初始人格设定用几个轻松选择题完成，不用填表单。可跳过、可在设置中回溯修改。
- **空状态不能是空白页**。没有记忆/待办/聊天记录时，用气泡提示引导用户："试试跟我说'帮我记一个待办'？"

### 9.2 状态过渡与微交互

- **状态之间不能直接跳变**，必须有过渡动画。紧急状态（错误、权限确认）可打断当前动画，其余状态排队处理。
- **连续相同状态合并去重**，避免桌宠"抽风"。
- **高唤醒状态需自动回退**：庆祝 3-5s 后自动回到待机，提醒 30s 无响应后降级。
- **微交互**：点击有弹性反馈，拖拽有惯性回弹，贴边自动吸附，悬停 0.5s 抬头看鼠标，双击快速聊天。

### 9.3 通知与气泡体验

- **气泡位置自适应屏幕**：桌宠在屏幕下半部分时气泡向上弹出，在上半部分向下弹出，确保不超出屏幕边界。
- **消息优先级**：紧急消息覆盖普通消息，最多同时显示 2 个气泡。
- **声音独立控制**：每种状态配短音效，音量不与系统绑定，右键一键静音。

### 9.4 主动互动设计

桌宠的"主动"是核心差异，但过度主动会变成骚扰。采用四级互动体系：

- **L0 纯视觉**：换姿势、眨眼、随机小动作，让桌宠"活着"，不产生任何通知。
- **L1 轻气泡**：偶尔关心（"该休息一下了"），不发声、不打断。
- **L2 标准提醒**：Agent 完成、待办到期，气泡 + 可选声音 + 可选系统通知。
- **L3 强提醒**：Agent 权限确认等紧急场景，持续提醒直到用户响应。

智能静默规则：
- P0：提供手动勿扰模式和一键隐藏气泡。
- P1：检测到全屏应用（游戏/视频/演示）时，自动降为 L0。
- P2：识别会议/录屏软件和屏幕共享场景，隐藏气泡和提醒内容。
- 用户可设定每日勿扰时段。
- 通过人格设置中的"主动程度"滑块控制 L1 是否开启。

### 9.5 错误与异常的用户感知

技术异常用人格化方式传达，不用技术语言：

- 网络断开 → 桌宠困惑："网络好像不太对。"
- 模型超时 → 桌宠皱眉："它在想……有点久。"
- 模型不可用 → 桌宠沮丧："AI 暂时回应不了，但你的待办和记忆都在。"

核心原则：**大模型离线时，本地功能（待办、记忆、Agent 监控、分享、通知）必须仍然可用**。异常恢复后桌宠主动告知。

### 9.6 分享体验流程

- **生成前必须预览**，绝不直接保存文件。预览窗口支持修改文案、选择模板、控制水印，确认后再保存或复制。
- **分享频率控制**：每日首次 Agent 完成主动提示分享一次，之后改为被动入口（右键菜单），避免骚扰。
- **一键复制**：保存和复制按钮并排，方便直接粘贴到社交媒体。

## 10. 结论

“赛博宇的桌面AI宠物”的机会点不只是做一个可爱的桌宠，而是做 **AI Agent 时代的桌面陪伴层**。

它应该先用个人形象和可爱动作吸引用户，再用文字聊天、基础记忆、待办提醒和 Agent 状态提醒形成真实价值，最后通过分享图、短视频、表情包和创作者生态完成传播与商业化。

首版应该坚决聚焦：

```text
桌宠形象
  + 文字聊天
  + 基础记忆
  + 待办提醒
  + Agent 状态提醒
  + 一键分享
```

只要这条链路跑通，这个项目就不只是一个玩具，而是一个能围绕 AI 工作流长出来的平台。
