# 赛博宇的桌面AI宠物（技术实现方案）

## 1. 技术目标

本项目优先面向 Windows，目标是做一个轻量、响应快、系统占用可控、扩展性好的桌面 AI 宠物。首版不追求极致性能，但必须避免常驻桌面应用常见的问题：内存占用过高、CPU 空转、动画无意义重绘、后台监听频繁轮询、模块耦合严重。

技术目标：

- Windows first，暂不考虑 macOS，Linux 可后续评估。
- 使用主流、资料多、适合 AI 辅助开发的技术栈。
- MVP 开发足够快，降低做不出来的风险。
- 系统架构高内聚、低耦合。
- 桌宠常驻时 CPU 和内存占用可接受。
- DeepSeek 不可用时，本地待办、提醒、动画、Agent 状态展示仍能工作。
- 从第一版开始保留插件化、记忆系统和 Agent Watcher 扩展接口。
- 第一版优先做出可演示、可分享、可验证用户兴趣的产品闭环。

## 2. 技术栈

### 2.1 总体选择

```text
桌面框架：Electron
前端：React + TypeScript + Vite
脚手架/构建：electron-vite
桌宠渲染：Canvas Sprite
本地存储：electron-store 起步，后续升级 SQLite
大模型：DeepSeek API
状态通信：Electron IPC + 内部事件总线
系统能力：Electron Main Process + Node.js
平台：Windows first
```

### 2.2 选择理由

#### Electron

Electron 不是最轻的桌面框架，但它最适合当前 MVP：

- AI 生成和调试 Electron 代码的成功率高。
- Windows 透明窗口、无边框窗口、置顶、托盘、通知、右键菜单都成熟。
- React、Canvas、DeepSeek 请求、分享图生成都能在 Web 技术栈里快速完成。
- TypeScript 学习成本高于 JavaScript，但能显著降低事件、IPC、插件、Agent 协议在 AI 辅助开发中的失控风险。
- 先做出产品闭环，再决定是否做更轻量的重构。

首版建议直接使用 electron-vite 的项目约定，而不是手写一套混合构建。它对 main、preload、renderer 三个入口有清晰约定，适合 React + Vite 项目快速启动。

#### React + TypeScript

首版使用 TypeScript，但采用“温和 TypeScript”策略：只把核心模型、事件、IPC、插件 manifest、AgentEvent、Todo、Reminder、Memory 等结构化数据类型化，不做复杂泛型和类型体操。

TypeScript 代码应遵守：

- 每个模块只做一件事。
- 核心数据结构必须有类型定义。
- 事件名集中管理。
- 不在 React 组件中堆业务逻辑。
- 不用 React state 驱动动画逐帧更新。
- 不为了“类型优雅”引入过度抽象。
- `strict` 可以分阶段开启，但新增核心模型必须写类型。

#### Canvas Sprite

首版不做 Live2D，不做 3D，不引入复杂渲染引擎。桌宠采用 sprite sheet + Canvas 绘制，满足像素风、低占用和快速开发。

选择原生 Canvas 而非 PixiJS 的理由：
- MVP 的像素精灵动画很简单（8-12 FPS 循环播放），Canvas 原生 API 完全够用，PixiJS 的 WebGL 渲染管线在这里是多余的。
- 避免引入额外依赖和包体积，减少构建复杂度。
- 如果后续动画复杂度升级（多层合成、滤镜、粒子效果），再引入 PixiJS。代码层面通过 Pet Engine 隔离渲染实现，上层只看到"播放状态 X 的动画"。

后续如果 Canvas 能力不够，再评估 PixiJS。

#### electron-store / SQLite

MVP 可以先用 electron-store 快速保存设置、待办和少量聊天记录。等数据结构稳定后，再迁移到 SQLite。

建议策略：

```text
MVP：electron-store
P1：SQLite
P2：加密、备份、同步
```

DeepSeek API Key 不应直接放进普通配置文件。首版优先用 Electron `safeStorage` 或操作系统凭据能力做安全存储；如果实现复杂，至少要避免写入日志、避免暴露到渲染进程。

## 3. 架构原则

### 3.1 Main 管系统，Renderer 管界面

Electron 有两个核心环境：

```text
Main Process：系统能力、窗口、托盘、通知、文件、Agent 监听
Renderer Process：React UI、桌宠动画、聊天气泡、待办面板、设置页
Preload Script：安全桥，只暴露白名单 API
```

Main Process 负责：

- 创建透明桌宠窗口。
- 创建设置窗口。
- 创建分享预览窗口。
- 托盘和右键菜单。
- 系统通知。
- 开机自启。
- 本地存储读写。
- DeepSeek 请求代理。
- Agent Watcher。
- 定时提醒调度。

Renderer Process 负责：

- 桌宠 Canvas 动画。
- 聊天气泡。
- 待办面板。
- 设置页。
- 分享预览。
- 用户交互。
- Pet Engine 状态展示。

Preload Script 负责：

- 暴露安全 API。
- 隔离 Node 能力。
- 转发事件。
- 不写业务逻辑。

### 3.2 开启 Electron 安全默认值

首版必须遵守这些安全约束：

```js
webPreferences: {
  contextIsolation: true,
  nodeIntegration: false,
  sandbox: true,
  preload: path.join(__dirname, 'preload.js')
}
```

原则：

- 渲染进程不能直接使用 Node.js。
- 不直接把 `ipcRenderer` 整个暴露给页面。
- preload 只暴露白名单 API。
- 所有从 renderer 到 main 的请求都要校验参数。
- 不加载远程页面作为主 UI。
- 不在日志中输出 API Key、Token、完整项目路径。

还需要加上这些窗口级保护：

```js
petWindow.webContents.setWindowOpenHandler(() => ({ action: 'deny' }));

petWindow.webContents.on('will-navigate', (event, url) => {
  if (!isAllowedNavigationUrl(url)) {
    event.preventDefault();
  }
});

session.defaultSession.setPermissionRequestHandler((_webContents, permission, callback) => {
  const allowed = ['notifications'].includes(permission);
  callback(allowed);
});
```

原则：

- 禁止任意新窗口。
- 禁止 renderer 导航到未知 URL。
- 权限请求默认拒绝，只对白名单能力放行。
- 所有外部链接用系统浏览器打开，不在桌宠窗口里打开。
- 生产环境配置 Content Security Policy，禁止任意脚本执行。

### 3.3 事件驱动，避免模块互相乱调

所有核心模块通过事件通信，不直接依赖彼此的内部实现。

示例事件：

```text
pet.state.change
chat.message.sent
chat.message.received
todo.created
todo.completed
todo.due
reminder.snoozed
agent.started
agent.running
agent.waiting_permission
agent.completed
agent.failed
agent.idle_too_long
memory.created
share.requested
share.generated
settings.updated
```

核心原则：

- Agent Watcher 不直接控制桌宠动画，只发事件。
- Todo Service 不直接操作 UI，只发提醒事件。
- DeepSeek Service 不直接写 React 状态，只返回结果或发事件。
- Pet Engine 只关心状态，不关心状态来源。

### 3.4 状态所有权原则

本项目采用“中心化事件通道 + 分布式状态所有权”，避免出现一个巨型全局状态管理器。

Event Bus 可以中心化，但它只负责：

- 发布事件。
- 订阅事件。
- 转发事件。
- 记录事件日志。
- 为调试面板和插件提供事件流。

Event Bus 不负责：

- 判断是否应该提醒用户。
- 判断桌宠应该播放什么动画。
- 判断待办是否过期。
- 判断 Agent 是否失败。
- 直接修改任何模块内部状态。

各模块拥有自己的状态：

```text
Pet Engine owns PetState
Todo Service owns TodoState
Reminder Service owns ReminderState
Agent Watcher owns AgentSessionState
Memory Service owns MemoryState
Chat Service owns ChatSessionState
Share Service owns ShareJobState
Settings Service owns SettingsState
Plugin Runtime owns PluginState
```

模块边界：

- Agent Watcher 不直接改 `PetState`，只发 `agent.*` 事件。
- Todo Service 不直接弹 UI，只发 `todo.*` 或 `reminder.*` 事件。
- Pet Engine 不关心事件来自 Claude、Trae、OpenClaw 还是待办，只根据事件优先级决定表现。
- Notification Service 不知道待办怎么存储，只消费提醒事件。
- Share Service 不直接监听原始 Agent 工具，只消费标准化后的 `AgentEvent`。
- Renderer store 只保存界面展示状态，不作为业务事实来源。

推荐事件流：

```text
Source Adapter
  -> Event Normalizer
  -> Event Bus
  -> State Owner / UI Consumer
```

示例：

```text
TraeAdapter
  -> AgentEventNormalizer
  -> agent.waiting_permission
  -> Event Bus
  -> PetEngine / NotificationService / ShareService
```

实现时禁止创建类似 `GlobalStateManager`、`AppStateGodObject` 的统一大状态对象。需要跨模块协作时，通过事件或明确的服务 API 完成。

### 3.5 本地功能不能依赖大模型

DeepSeek 只提供智能能力：

- 文字聊天。
- 自然语言创建待办。
- 记忆提取建议。
- 分享文案生成。
- Agent 结果总结。

以下能力必须离线可用：

- 桌宠显示和动画。
- 待办查看、完成、删除。
- 定时提醒。
- Agent 状态接收和展示。
- 本地记忆查看和删除。
- 分享图基础生成。

DeepSeek 不可用时的降级行为：

| 功能 | DeepSeek 在线 | DeepSeek 离线 |
|------|-------------|-------------|
| 文字聊天 | 正常 | 禁用，气泡提示"网络未连接" |
| 自然语言创建待办 | 正常 | 降级为手动输入 |
| 智能记忆建议 | 正常 | 关闭，仅保留用户手动记忆 |
| 分享文案生成 | 正常 | 使用预设模板文案 |
| Agent 结果总结 | 正常 | 显示原始状态信息 |

DeepSeek Service 主进程实现要点：
- 请求超时 30s，超时后返回降级提示。
- 连续失败 3 次后自动暂停，由用户手动重试。
- 通过监听系统网络状态事件判断离线/恢复，恢复后主动通知桌宠。

### 3.6 MVP 不做过度平台化

首版只做“内置插件架构”，不做开放插件市场。插件系统先用于拆分内部能力，避免代码变成一团。

首批内置插件：

- `todo`
- `agent-watcher`
- `memory`
- `share`
- `deepseek`

## 4. 系统模块

### 4.1 App Shell

职责：

- 创建透明桌宠窗口。
- 创建设置窗口。
- 创建分享预览窗口。
- 管理系统托盘。
- 管理开机自启。
- 管理全局快捷入口。
- 管理窗口显示、隐藏、退出。

窗口策略：

- 桌宠窗口默认透明、置顶、无边框。
- 设置页和分享页使用普通窗口。
- 桌宠窗口尽量小，只覆盖桌宠和气泡区域。
- 支持拖拽移动和屏幕边缘吸附。
- 支持勿扰模式。
- P0：提供手动勿扰模式和一键隐藏气泡。
- P1：检测到全屏窗口后降低主动打扰。
- P2：识别会议/录屏软件和屏幕共享场景，隐藏气泡和提醒内容。

### 4.2 Pet Engine

职责：

- 管理桌宠状态。
- 管理动画播放。
- 管理状态优先级。
- 管理主动打扰等级。
- 接收外部事件并切换表现。

核心状态：

```ts
export type PetState =
  | 'idle'
  | 'working'
  | 'reminding'
  | 'success'
  | 'error'
  | 'waiting'
  | 'sleeping'
  | 'sharing';
```

状态优先级：

```text
error / waiting_permission
  > reminder_due
  > agent_completed
  > working
  > idle
```

动画策略：

- 待机动画 8-12 FPS。
- 响应动画 24 FPS。
- 无状态变化时不触发 React 组件重渲染。
- Canvas 自己控制逐帧绘制。
- 窗口隐藏、全屏应用、勿扰模式下暂停或降帧。
- sprite sheet 预加载，避免状态切换时卡顿。

主动互动分级：

```text
L0 纯视觉：idle 循环 + 随机微动，不产生通知
L1 轻气泡：偶尔关心，不发声、不打断
L2 标准提醒：Agent 完成、待办到期，气泡 + 可选声音 + 可选通知
L3 强提醒：Agent 权限确认，持续提醒直到用户响应
```

状态过渡与队列：

- 状态之间不直接跳变，通过过渡动画衔接。
- 维护状态队列，当前动画未完成时排队处理，紧急状态（error、waiting_permission）可插队。
- 连续相同状态合并去重。
- 庆祝状态 3-5s 后自动回退到 idle，提醒 30s 无响应降级。

### 4.3 Chat Service

职责：

- 处理用户输入。
- 调用 DeepSeek。
- 展示桌宠气泡回复。
- 识别是否需要创建待办、记忆或触发插件。

首版支持：

- DeepSeek API Key 配置。
- 普通文本聊天。
- 聊天失败降级提示。
- 从自然语言中识别待办和提醒。

DeepSeek 请求建议放在 Main Process 中，由 Renderer 通过白名单 API 调用：

```text
renderer chat input
  -> preload api.chat.sendMessage
  -> main deepseekService
  -> DeepSeek API
  -> sanitized response
  -> renderer
```

这样可以避免 API Key 暴露到前端。

### 4.4 Todo / Reminder Service

职责：

- 创建待办。
- 完成待办。
- 删除待办。
- 管理提醒时间。
- 触发桌宠提醒事件。
- 触发系统通知。

提醒策略：

- Main Process 负责定时任务，避免前端窗口隐藏后提醒失效。
- Renderer 只负责展示提醒和用户操作。
- 支持完成、稍后提醒、忽略。
- 应用退出后是否继续提醒，MVP 可以先不做，后续可用开机自启和本地任务恢复补齐。

### 4.5 Agent Watcher

职责：

- 监听 AI Agent 状态。
- 标准化不同 Agent 的状态事件。
- 通知 Pet Engine 切换状态。
- 通知 Share Service 生成分享素材。

P0 支持对象：

- Agent 状态模拟器。
- 手动“帮我盯这个任务”兜底入口。

P0.5 / 内测增强支持对象：

- Claude Code / Codex / Trae / 小龙虾 OpenClaw 中选择一个状态来源最稳定的工具先打通。
- 普通终端任务作为兜底。

状态协议：

```ts
export type AgentStatus =
  | 'started'
  | 'running'          // 包含"正在思考"，通过 message 补充区分
  | 'waiting_permission'
  | 'completed'        // "生成了关键文件"作为 completed 的附带信息，通过 message 传递
  | 'failed'
  | 'idle_too_long';

export type AgentKind =
  | 'claude-code'
  | 'codex'
  | 'trae'
  | 'openclaw'
  | 'terminal';

export interface AgentEvent {
  id: string;
  type: 'agent.started' | 'agent.running' | 'agent.waiting_permission' | 'agent.completed' | 'agent.failed' | 'agent.idle_too_long';
  source: 'agent-watcher';
  agent: AgentKind;
  sessionId: string;
  status: AgentStatus;
  message?: string;
  projectPath?: string;
  priority: 'low' | 'normal' | 'high';
  createdAt: string;
}

// 运行时 JSON 序列化使用 snake_case，与产品文档协议保持一致：
// { "id": "agent_event_001", "type": "agent.waiting_permission", "agent": "...", "session_id": "...", "status": "...", "project_path": "...", "created_at": "..." }
```

监听优先级：

```text
插件主动上报
  > 终端桥接
  > 日志监听
  > 进程/窗口低频检测
```

P0 先做 Agent 状态模拟器，跑通体验后再接真实工具。真实 Agent Watcher 属于 P0.5 / 内测增强，不阻塞 P0。

终端桥接具体思路（优先打通 Claude Code）：

- 候选方案：验证 Claude Code 是否存在稳定 history/log/jsonl 文件；只有确认稳定后，才作为 adapter 实现依据。
- 备选方案：监听终端窗口标题变化（如 "Claude Code — Waiting for permission"），或监控项目目录下 `.claude/` 的文件变更频率。
- 兜底方案：低频进程检测（每分钟一次检查 claude 进程是否存在），用于判断 started/completed。

### 4.6 Memory Service

P0 只做最简记忆存储（支持"记住 X"/"忘记 X"命令，electron-store 存数组，最多 20 条），不在此阶段耽误主线。

P1 实现完整记忆管理页（查看、搜索、编辑、删除、按类型筛选）和插件建议记忆确认流程。

### 4.7 Share Service

职责：

- 生成分享图。
- 预览分享内容。
- 做基础隐私过滤。
- 保存 PNG。
- 复制文案。

首版只做静态分享图，不做 GIF/MP4。

隐私过滤：

- API Key。
- Token。
- 邮箱。
- 手机号。
- 用户名。
- 绝对路径。
- 项目敏感文件名。

## 6. MVP 开发顺序

### 6.1 第 1 步：桌宠窗口

- 初始化 Electron + React + TypeScript + Vite。
- 创建透明、无边框、置顶窗口。
- 显示默认桌宠图片。
- 支持拖拽移动。
- 支持右键菜单。
- 支持托盘退出。

验收标准：

- Windows 上能启动。
- 桌宠能置顶显示。
- 鼠标拖拽不卡顿。
- 关闭后托盘仍可退出。

### 6.2 第 2 步：Canvas 动画

- 加载 sprite sheet。
- 实现 idle 动画。
- 实现 working、success、error 状态。
- React 只传状态，Canvas 自己绘制。

验收标准：

- idle 动画常驻时 CPU 占用低。
- 切换状态无明显卡顿。
- 隐藏窗口或勿扰模式下降帧/暂停。

### 6.3 第 3 步：聊天气泡 + DeepSeek

- 点击桌宠弹出聊天气泡。
- 配置 DeepSeek API Key。
- 发送消息并展示回复。
- 失败时给本地降级提示。

验收标准：

- DeepSeek 可用时能聊天。
- DeepSeek 不可用时不会影响桌宠和待办。
- API Key 不出现在前端日志中。

### 6.4 第 4 步：待办与提醒

- 创建待办。
- 完成待办。
- 设置提醒时间。
- 到点触发桌宠气泡。
- 到点触发系统通知。

验收标准：

- 应用隐藏后提醒仍能触发。
- 提醒可完成、稍后提醒、忽略。

### 6.5 第 5 步：Agent 状态模拟器

- 做一个开发调试面板。
- 手动触发 started、waiting_permission、completed、failed。
- Pet Engine 根据事件切换状态。

验收标准：

- Agent 状态变化能快速反映到桌宠动作和气泡。
- waiting_permission 优先级高于普通提醒。

### 6.6 第 6 步：P0.5 真实 Agent Watcher

- 先选择一个工具打通完整链路。
- 优先考虑能稳定获取状态的方式。
- 状态全部转换为标准 AgentEvent。
- 真实 Agent Watcher 不阻塞 P0 发布；如果集成卡住，先保留模拟器和手动兜底入口。

验收标准：

- 能稳定识别运行、等待确认、完成、失败中的至少 3 类。
- 不依赖高频屏幕识别。

### 6.7 第 7 步：分享图

- Agent 完成后出现分享入口。
- 生成分享卡片预览。
- 做基础隐私过滤。
- 保存 PNG。

验收标准：

- 分享图可读、好看、不泄露敏感路径。
- 用户确认后才保存。

## 7. 性能约束

### 7.1 常驻资源目标

MVP 阶段建议目标：

```text
空闲 CPU：尽量 < 2%
常驻内存：尽量 < 250MB
待机动画：8-12 FPS
响应动画：24 FPS
Agent 状态变化响应：< 500ms
普通提醒触发延迟：< 1s
```

Electron 常驻内存很难做到极低，首版目标是“用户可接受”，不是极致轻量。

### 7.2 性能原则

- 不用 React state 驱动每一帧动画。
- 不做高频进程扫描。
- 不做屏幕 OCR 级别状态识别。
- 不在主线程做大图生成。
- 不在前端长期保存大对象。
- 不在每次聊天时塞入全部历史。
- 不让 DeepSeek 请求阻塞 UI。
- P0 支持手动勿扰降级；P1/P2 再支持全屏、投屏自动降级。
- 尽量只保留一个常驻渲染窗口，设置和分享窗口按需创建。
- 对 Agent 监听使用事件或低频检测，禁止无意义 busy loop。

### 7.3 性能监控

开发阶段加入简易性能面板，避免性能目标变成空口号：

- Main Process 每 30s 记录一次内存和 CPU 占用，输出到日志文件。
- 开发模式下在桌宠右键菜单中增加"性能面板"入口，实时显示当前 FPS、内存、CPU。
- smoke test 中加入：启动后 idle 状态下 CPU 不超过 x%。

## 8. Electron 安全与 IPC 规范

### 8.1 BrowserWindow 默认配置

```js
const petWindow = new BrowserWindow({
  width: 360,
  height: 360,
  transparent: true,
  frame: false,
  alwaysOnTop: true,
  skipTaskbar: true,
  resizable: false,
  webPreferences: {
    preload: path.join(__dirname, '../preload/index.js'),
    contextIsolation: true,
    nodeIntegration: false,
    sandbox: true
  }
});
```

说明：源码使用 `preload/index.ts`，构建后产物路径由 electron-vite 输出为 JavaScript 文件。文档示例中的 `index.js` 指生产运行路径，不代表源码使用 JavaScript。

### 8.2 Preload 只暴露白名单 API

```ts
contextBridge.exposeInMainWorld('deskPet', {
  chat: {
    sendMessage: (input) => ipcRenderer.invoke('chat:send-message', input)
  },
  todo: {
    create: (input) => ipcRenderer.invoke('todo:create', input),
    list: () => ipcRenderer.invoke('todo:list'),
    complete: (id) => ipcRenderer.invoke('todo:complete', id),
    delete: (id) => ipcRenderer.invoke('todo:delete', id)
  },
  reminder: {
    list: () => ipcRenderer.invoke('reminder:list'),
    snooze: (id) => ipcRenderer.invoke('reminder:snooze', id)
  },
  memory: {
    list: (type?: string) => ipcRenderer.invoke('memory:list', type),
    create: (input) => ipcRenderer.invoke('memory:create', input),
    delete: (id) => ipcRenderer.invoke('memory:delete', id)
  },
  agent: {
    getStatus: () => ipcRenderer.invoke('agent:get-status'),
    getHistory: () => ipcRenderer.invoke('agent:get-history')
  },
  share: {
    generate: (input) => ipcRenderer.invoke('share:generate', input),
    preview: (input) => ipcRenderer.invoke('share:preview', input)
  },
  settings: {
    getPetConfig: () => ipcRenderer.invoke('settings:get-pet-config'),
    updatePetConfig: (input) => ipcRenderer.invoke('settings:update-pet-config', input),
    getNotificationConfig: () => ipcRenderer.invoke('settings:get-notification-config'),
    updateNotificationConfig: (input) => ipcRenderer.invoke('settings:update-notification-config', input)
  },
  events: {
    onPetEvent: (callback) => {
      const listener = (_event, payload) => callback(payload);
      ipcRenderer.on('pet:event', listener);
      return () => ipcRenderer.removeListener('pet:event', listener);
    }
  }
});
```

完整 IPC 通道名和事件名统一在 `shared/eventNames.ts` 和 `shared/ipcChannels.ts` 中集中管理，新增通道必须先在 shared 中注册。

禁止：

- 暴露完整 `ipcRenderer`。
- 在 renderer 中读取文件系统。
- 在 renderer 中持有 DeepSeek API Key。
- 使用远程代码执行能力。

### 8.3 IPC 参数校验

所有 `ipcMain.handle` 都必须校验输入：

```ts
ipcMain.handle('todo:create', async (_event, input) => {
  const todo = validateCreateTodoInput(input);
  return todoService.create(todo);
});
```

MVP 可以先手写校验函数，后续再引入 Zod 等 schema 校验库。TypeScript 类型不能替代运行时校验，所有来自 renderer、文件、插件和外部进程的数据都要校验。

### 8.4 Content Security Policy

生产环境必须配置 CSP，防止任意脚本执行。首版可以从保守策略开始：

```html
<meta
  http-equiv="Content-Security-Policy"
  content="default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self';"
/>
```

开发环境可以为 Vite dev server 放宽，但生产包必须收紧。

### 8.5 依赖和远程内容

- Electron、Vite、React 等核心依赖保持较新版本。
- 不使用已经停止维护的 Electron 模板。
- 不把远程网页嵌入主窗口。
- 不允许 renderer 动态执行远程脚本。
- 每次发布前跑依赖审计。

## 9. 插件架构

首版插件只用于内部模块化，不开放第三方安装。

插件 Manifest 示例：

```json
{
  "id": "agent-watcher",
  "name": "Agent Watcher",
  "version": "0.1.0",
  "permissions": [
    "process:read",
    "window:read",
    "notification:send",
    "pet:react",
    "memory:suggest"
  ],
  "events": [
    "agent.started",
    "agent.waiting_permission",
    "agent.completed",
    "agent.failed"
  ]
}
```

权限原则：

- 插件必须声明权限。
- 敏感权限必须在设置页可见。
- 插件建议记忆必须用户确认。
- 插件不能直接操作 UI，只能发事件。
- 内置插件也按权限模型开发，方便后续开放生态。

## 10. 数据存储

### 10.1 MVP 数据

MVP 使用 electron-store 保存普通本地数据，但只允许 Main Process 直接访问存储。Renderer 通过 preload 白名单 API 读写，不直接碰本地文件。

```text
settings
todos
reminders
conversations
messages
memories
agent_sessions
agent_events
share_artifacts
plugin_settings
```

### 10.2 P1 升级 SQLite

当以下情况出现时，迁移到 SQLite：

- 聊天记录明显增多。
- 需要按时间、类型、标签查询记忆。
- Agent 事件需要统计。
- 分享记录需要检索。
- 待办和提醒需要更稳定的数据关系。

### 10.3 本地数据原则

- 默认本地保存。
- 用户可清空聊天、待办、记忆、Agent 日志。
- 分享图生成前必须预览。
- DeepSeek API Key 需要安全存储，不明文暴露到前端日志。
- 分享前必须做基础脱敏。
- electron-store 只保存非敏感配置和业务数据。
- API Key、Token 等敏感数据优先放入系统安全存储。

## 13. 风险与对策

### 13.1 Electron 常驻占用偏高

风险：Electron 的基础内存占用高于原生方案。

对策：

- 只保留必要窗口。
- 设置页、分享页按需创建和销毁。
- Canvas 控制帧率。
- 无状态变化不重绘。
- 禁止后台 busy loop。
- 先用性能指标约束 MVP，不追求极致轻量。

### 13.2 TypeScript 复杂度过高

风险：TypeScript 使用过度后，出现复杂泛型、类型体操和过度抽象，拖慢 MVP。

对策：

- 只把核心模型、IPC、事件和插件协议类型化。
- 事件名集中在 `shared/eventNames.ts`。
- 公共类型集中在 `shared/types.ts`。
- 不写复杂泛型。
- 不为了复用提前抽象。
- 所有 IPC 入口做参数校验。
- 每个服务只暴露少量方法。
- 每次 vibe coding 只改一个模块。

### 13.3 巨型全局状态失控

风险：为了开发方便，把宠物、聊天、待办、提醒、Agent、记忆、分享、设置全部塞进一个全局 store，后续任何功能都互相影响。

对策：

- Event Bus 只做事件通道，不做业务判断。
- 每个模块拥有自己的状态。
- Renderer store 只存界面展示状态。
- 业务事实来源在对应 service 中。
- 跨模块协作用事件或明确服务 API，不直接读写其他模块内部状态。
- code review 时重点检查是否出现 `globalStore`、`appState`、`stateManager` 这类过度集中的对象。

### 13.4 Agent 状态不稳定

风险：不同 Agent 工具没有统一状态 API。

对策：

- 先用模拟器跑通体验。
- 再选一个工具打通真实集成。
- 保留手动“帮我盯这个任务”的兜底入口。
- 所有状态都转换为标准 AgentEvent。

### 13.5 安全边界被打穿

风险：为了开发方便打开 `nodeIntegration` 或暴露完整 IPC，后续很难修。

对策：

- 从第一天启用 `contextIsolation`。
- 从第一天关闭 `nodeIntegration`。
- 从第一天使用 preload 白名单 API。
- 从第一天把 DeepSeek 请求放 Main Process。

## 14. GitHub 与官方实践对齐结论

调研成熟 Electron 项目和官方安全实践后，本方案做以下取舍：

- 使用 electron-vite 风格的 `main / preload / renderer` 三入口结构，减少自定义构建复杂度。
- 源码使用 TypeScript，但保持温和策略，避免复杂类型拖慢 MVP。
- 使用 preload + contextBridge 暴露白名单 API，不把 Node 能力交给 renderer。
- 默认关闭 `nodeIntegration`，开启 `contextIsolation` 和 `sandbox`。
- Main Process 负责 DeepSeek、存储、通知、Agent 监听等系统能力。
- Renderer 只做 UI 和 Canvas 动画。
- Event Bus 中心化，业务状态分散到各自模块 ownership，避免巨型全局状态。
- 配置 CSP、导航拦截、外部窗口拦截、权限白名单。
- 首版使用内置插件架构，不开放第三方插件安装。
- 加入 smoke test 和 Windows 打包检查，避免 MVP 后期不可控。

参考来源：

- [electron-vite Development](https://electron-vite.org/guide/dev.html)：参考其 `main / preload / renderer` 目录约定，以及避免在 renderer 使用 Node.js 模块的建议。
- [electron-vite Building for Production](https://electron-vite.org/guide/build.html)：参考 `electron-vite build` 的生产构建流程。
- [Electron Security](https://www.electronjs.org/docs/latest/tutorial/security)：参考 Electron 官方安全清单。
- [Electron Context Isolation](https://www.electronjs.org/docs/latest/tutorial/context-isolation)：参考 `contextBridge` 和 preload 白名单 API 模式。
- [electron-react-boilerplate](https://github.com/electron-react-boilerplate/electron-react-boilerplate)：参考成熟 Electron + React 项目的主进程、渲染进程、构建和发布组织方式。
- [secure-electron-store](https://github.com/reZach/secure-electron-store)：参考“存储通过 IPC 暴露，不让 renderer 直接操作本地存储”的方向。

## 15. 总结

本项目 MVP 技术路线确定为：

```text
Electron + React + TypeScript + Vite
```

核心思想：

```text
Electron Main 负责系统能力
React 负责界面
TypeScript 负责核心模型和协议约束
Canvas 负责桌宠动画
Preload 负责安全桥
事件总线负责模块解耦
DeepSeek 负责智能能力
electron-store 负责 MVP 本地数据
```

首版最重要的不是一次性做完整平台，而是跑通：

```text
桌宠常驻
  + DeepSeek 聊天
  + 待办提醒
  + Agent 状态提醒
  + 分享图
```

只要这条链路稳定，后续再扩展记忆、插件、更多 Agent 和创作者生态。
